# Outline Agent — 小说大纲生成器设计

**日期**: 2026-07-06
**状态**: 设计已确认，待实现

## 问题

当前"剧本 agent"（`runOutlineWriter`）源于影视剧行业的"场景节拍"（Scene Beats）概念——为单个场景生成 3-5 个情节节拍。但小说写作中，决定"写什么"的不是"场景节拍"，而是**大纲**。

## 大纲 vs 剧本

| | 剧本 (Screenplay Beat) | 大纲 (Novel Outline) |
|---|---|---|
| 颗粒度 | 场景级（scene-level） | 章节级（chapter-level） |
| 回答的问题 | "这个场景怎么拍" | "这一章要写什么" |
| 产出物 | 3-5 个节拍 + 情感弧线 | 章节结构 + 情节要点 + 角色线索 + 伏笔布局 |
| 行业来源 | 好莱坞编剧 | 小说创作 |

## 大纲 Agent

### 它回答的核心问题

"承接第 N 章，下一章应该写什么？"

### 它需要知道什么

1. **前情** — 前文摘要、最近几章发生了什么、当前角色状态
2. **角色** — 谁还活着、在哪、想干什么、彼此关系如何
3. **伏笔** — 哪些伏笔还悬着、哪些到了回收窗口
4. **世界观** — 当前场景可选的地点、势力分布、规则约束
5. **作者意图** — 用户可选的指导（比如"这一章要推进感情线"、"要有战斗场景"）

### 它输出什么

- 章节标题/主题
- 章节目标（这一章要达成什么）
- 3-5 个关键情节点（比"节拍"更粗粒度——不是"角色A打开门"，而是"角色A发现了关于B的秘密"）
- 涉及的角色和他们的线索
- 建议埋入的新伏笔
- 建议回收的旧伏笔
- 情感弧线
- 节奏建议

### 是否需要工具调用能力

**不需要。** 大纲 agent 是一次性生成，不是多轮迭代。

理由：大纲 agent 的信息全部可以在 prompt 中注入——前情摘要、角色状态、伏笔账本、世界设定都在 Codex 中。1M 上下文窗口足够容纳所有这些。大纲不需要"先查再写"——所有的"查"在 prompt 组装时已完成。

### 与 Writer Agent 的关系

```
用户选择承接章节 → Outline Agent 生成大纲 → 用户审阅/编辑大纲 → Writer Agent 根据大纲创作正文 → Review Agents 审查
```

大纲 agent 的输出成为 Writer agent 的 Codex 第 4 段（前文摘要层）和第 7 段（当前任务）的输入。

## 实现

### 修改

- `src/core/simulation/director.ts` → 重命名为 `outline-agent.ts`
- `runOutlineWriter()` → 重命名为 `generateOutline()`
- System prompt 重写为小说大纲生成器

### 大纲 Agent 的 System Prompt 要点

- 角色定位：有经验的小说大纲师，不是编剧
- 理解承接关系：明确知道从哪一章/哪个情节继续
- 关注角色弧线：不是独立的场景节拍，而是角色成长的连续性
- 关注伏笔：主动建议推进或回收已有伏笔
- 输出结构化 JSON（保持 OUTLINE_SCHEMA）
