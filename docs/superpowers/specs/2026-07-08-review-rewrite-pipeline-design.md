# Automated Review-Rewrite Pipeline Design

**Date:** 2026-07-08
**Status:** draft

## Problem

The current review flow has three critical issues:

1. **Review findings are invisible.** The `ReviewFinding` has `snippet` (problematic text) and `fixedText` (correction), but neither is shown in the UI. Users see a wall of severity badges with no way to understand what was actually changed.
2. **"Auto-fix" is fake.** `fixedText` is stored in JSON but never applied to the prose. The displayed prose is always the unrevised draft.
3. **Review is a dead-end report.** There is no rewrite step — findings are thrown at the user with no automated resolution.

## Goal

Turn review into a quality gate: prose passes through review, issues are fed back to the writer for a full rewrite, and the final corrected version is what the user sees. Inline annotations show what was changed and why.

## Design

### 1. New Pipeline

```
Writer generates draft prose
  → 6 review agents run in parallel (character, continuity, foreshadowing, style, world, pacing)
  → Findings are merged
  → Rewrite agent (LLM) receives: original prose + all findings (snippets + suggestions)
  → Rewrite agent produces corrected full prose
  → Position-matching generates ProseAnnotation[]
  → Final prose + annotations displayed to user
  → scene_end
```

### 2. SSE Event Stream

Events emitted in order:

| Event | When | Data |
|-------|------|------|
| `prose` | Writer finishes draft | `{ prose: string }` |
| `review` | Review agents running | `{ status: "reviewing" }` (loading indicator) |
| `rewriting` | Rewrite in progress | `{ status: "rewriting" }` (loading indicator) |
| `final_prose` | Rewrite complete | `{ prose: string, annotations: ProseAnnotation[] }` |
| `scene_end` | Pipeline done | `{ fullNovel: string }` (unchanged) |

### 3. Types

```typescript
interface ProseAnnotation {
  id: string;
  finding: ReviewFinding;
  originalSnippet: string;   // flagged text from draft
  fixedSnippet: string;       // corrected text in final prose
  position: {
    start: number;            // character offset in final prose
    end: number;
  };
}
```

`ReviewFinding` remains unchanged — `snippet` and `fixedText` already exist on it.

### 4. Rewrite Agent

A single LLM call with structured output:

**Input:**
- Original prose (full text)
- All findings from all 6 review agents: `[{ dimension, severity, snippet, suggestion, description }, ...]`

**System prompt (conceptual):**
```
You are a prose rewriter. Given the original prose and review findings, rewrite the prose to fix ALL issues. Preserve the narrative flow, style, and pacing of the original. Do not change content unrelated to the findings. Return the complete corrected prose.
```

**Output:** Complete corrected prose (string).

The rewrite is one LLM call, not multiple — same as the original writer call.

### 5. Annotation Generation

After the rewrite produces corrected prose, annotations are generated by matching `snippet` from each finding against the draft, and `fixedText` from each finding against the final prose:

- For each finding where `autoFixable === true` and `fixedText` exists: search for `snippet` in draft prose to determine original position, search for `fixedText` in final prose to determine corrected position
- For findings where `autoFixable === false`: include finding but `fixedSnippet` is empty (marks issues that need human attention)
- Fallback: if exact string matching fails, annotate at chapter level (position covers the whole section)

This matching runs server-side before the `final_prose` event is emitted.

### 6. UI Changes

**Right panel — prose display:**

- Shows `final_prose.prose` as the canonical text (NOT the draft)
- Annotated regions have inline highlight styling:
  - Critical: red underline
  - Major: orange underline
  - Minor: subtle dashed underline
- Hover on annotation: tooltip popup showing:
  - Dimension label (character/continuity/...)
  - Severity badge
  - Original text (from draft)
  - Corrected text (from final)
  - Why (description from finding)

**Right panel — review summary:**

The existing ReviewSection is reworked into a collapsible "变更摘要" (change summary):
- Header: "审查修正完成 — N 处修改 (M 处待确认)"
- Expandable list of all annotations, grouped by dimension
- Each entry shows: dimension icon, one-line description, severity badge
- Click scrolls the reader to that annotation's position

**Save behavior:**

"保存为最新章节" saves `final_prose.prose` (the corrected version). The draft is discarded on save.

### 7. Engine Changes

**`src/core/simulation/engine.ts`:**

After the review step in `run()`:
1. Collect all findings from `ReviewReport`
2. If any auto-fixable findings exist, call rewrite agent
3. Generate annotations by matching snippets
4. Emit `final_prose` event
5. Use `final_prose.prose` for codex updates (updateCodexAfterChapter) and `scene_end`

**`src/core/codex/review-orchestrator.ts`:**

Add `rewriteProse()` function:
- Input: `(originalProse: string, findings: ReviewFinding[], codex: WritersCodex)`
- Calls LLM with findings + original prose
- Returns corrected prose string

**`src/core/simulation/types.ts`:**

Add `final_prose` and `rewriting` to `SimulationEvent`:

```typescript
| { type: "rewriting"; status: string }
| { type: "final_prose"; prose: string; annotations: ProseAnnotation[] }
```

### 9. Edge Cases

- **No findings:** Skip rewrite entirely. Emit `final_prose` with empty annotations. No visual change from current flow.
- **Rewrite introduces new issues:** Not checked in this version. A follow-up could add a "review-the-rewrite" step.
- **Snippet matching fails:** Fall back to chapter-level annotation (no inline highlight, just a note in the summary).
- **Very long chapters:** Snippets from beyond 8000 chars may not have agent coverage. Annotate that those findings apply to "chapter beyond review window."

### 10. Out of Scope

- Reviewing the rewrite itself (meta-review loop)
- User manually editing the corrected prose before save
- Per-paragraph rewrite (always full text)
- Reducing the number of review agents
